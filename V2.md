# Disaggregated Inference Orchestrator

## Product Document

### Author

Lekan — Senior Software Engineer

### Date

March 2026

---

## 1. Problem Statement

Modern LLM inference has two fundamentally different computational phases running on the same GPU:

**Prefill** processes the entire input prompt in parallel. It is compute-bound — the GPU's math units are the bottleneck. It generates the KV cache and the first output token.

**Decode** generates output tokens one at a time, autoregressively. It is memory-bound — the GPU spends most of its time streaming model weights from HBM to compute cores, not doing math.

In standard serving, both phases share the same GPU. This creates interference: a large prefill from a new request blocks token generation for users already mid-conversation, causing latency spikes. The GPU is never optimally utilized for either phase because it constantly switches between two workload profiles with opposite hardware requirements.

---

## 2. What We're Building

An inference orchestration layer that separates prefill and decode into dedicated GPU pods, with an inference-aware Kubernetes GPU scheduler managing placement, a custom KV cache transfer mechanism coordinating the handoff between phases, and adaptive serving logic that dynamically chooses the optimal serving strategy based on real-time traffic.

The system uses vLLM as the underlying inference engine for model loading, attention computation, and token generation. We build the disaggregation layer on top: the prefill service, the decode service, the KV cache transfer, the request router, the adaptive autoscaler, and the scheduler integration.

---

## 3. Goals

**Primary:** Demonstrate that disaggregated serving achieves higher throughput than standard serving under concurrent load, with real benchmarks across two hardware tiers.

**Secondary:** Extend the existing GPU scheduler with workload-aware scheduling capabilities — understanding prefill versus decode workload profiles and placing pods accordingly based on GPU topology and resource availability.

**Tertiary:** Produce a portfolio-quality project that demonstrates end-to-end inference infrastructure engineering: scheduling, memory management, GPU communication, and performance benchmarking.

## 4. Architecture

### Components

**Request Router**
Lightweight service that receives inference requests from clients. Assigns each request a unique ID. Forwards prompts to an available prefill pod. Once prefill completes, routes the KV cache transfer metadata to a decode pod. Streams generated tokens back to the client from the decode pod.

**Prefill Service**
A pod running on the designated prefill GPU. Receives the full prompt, runs the forward pass through all transformer layers via vLLM, generates the KV cache and the first token. Does not continue decoding. Signals the router that the KV cache is ready for transfer. Optimized for batching multiple concurrent prefills since the workload is compute-bound.

**Decode Service**
A pod running on the designated decode GPU. Receives the transferred KV cache, loads it into GPU memory, and generates tokens autoregressively via vLLM until the response is complete. Streams tokens back to the client through the router. Handles many concurrent decode sessions since each is memory-bound and individually light on compute.

**KV Cache Transfer Layer**
Moves the KV cache from the prefill GPU to the decode GPU after prefill completes. On A100 SXM4 with NVLink, uses NCCL for direct GPU-to-GPU transfer at ~600 GB/s. On RTX 4090 over PCIe, transfers through CPU memory at ~24 GB/s. Measures transfer time precisely as it is the primary overhead of disaggregation.

**GPU Scheduler (existing, extended)**
The Kubernetes-native GPU scheduler, extended with workload-aware placement and adaptive serving modes. Understands pod labels for inference phase (prefill vs decode). Places pods on GPUs based on topology awareness — preferring NVLink-connected pairs for prefill and decode co-location. Monitors real-time metrics (queue depths, GPU utilization, TTFT, ITL) and dynamically rebalances GPU allocation between prefill and decode pods. Manages pod lifecycle, scaling, preemption, and serving mode transitions.

**Adaptive Autoscaler**
A control loop within the scheduler that monitors traffic patterns and decides the optimal serving strategy in real time. Watches prefill queue depth, decode queue depth, per-GPU utilization, TTFT, and ITL. Triggers GPU reallocation when metrics breach configured thresholds. Manages transitions between serving modes (see Section 4a below) with minimal disruption to in-flight requests.

**Benchmarking Harness**
Sends concurrent requests at configurable rates. Measures per-request TTFT, tokens per second (TPS), inter-token latency (ITL), end-to-end latency, overall throughput (requests per second), and KV cache transfer time. Runs identical workloads against both disaggregated and baseline deployments.

### Request Flow

1. Client sends prompt to the request router.
2. Router assigns the request to an available prefill pod.
3. Prefill pod runs the full forward pass via vLLM, produces KV cache and first token.
4. KV cache transfers from prefill GPU to decode GPU via NCCL (NVLink) or CPU staging (PCIe).
5. Decode pod loads the KV cache, generates tokens one by one via vLLM.
6. Tokens stream back through the router to the client.

### Adaptive Serving Modes

The scheduler operates in one of three modes based on real-time traffic, transitioning automatically as conditions change.

**Unified Mode (low traffic)**
All GPUs run standard vLLM serving — each GPU handles both prefill and decode. No disaggregation overhead, no KV cache transfer. This is the default mode at startup and during low-traffic periods. Remaining GPUs can be released for other workloads or idled to save cost.

Triggers to enter: concurrency drops below the crossover threshold where disaggregation adds overhead without throughput benefit.

**Static Disaggregated Mode (medium traffic)**
GPUs are split into dedicated prefill and decode pools with a fixed ratio (e.g., 2 prefill, 2 decode on the 4x 4090 setup). Disaggregation is active. KV caches transfer between pools. This mode delivers the throughput gains of disaggregation with predictable, stable resource allocation.

Triggers to enter: concurrency exceeds the crossover threshold and traffic is stable.

**Dynamic Disaggregated Mode (high or variable traffic)**
The scheduler actively rebalances GPUs between prefill and decode pools based on real-time demand. During a spike of new users, the scheduler shifts GPUs toward prefill (e.g., 3 prefill, 1 decode). When the spike settles into long conversations, it shifts back toward decode (e.g., 1 prefill, 3 decode).

Triggers to rebalance:

* TTFT degrading beyond target → need more prefill capacity.
* ITL degrading beyond target → need more decode capacity.
* Prefill queue depth growing → need more prefill capacity.
* Decode queue depth growing → need more decode capacity.

The rebalancing process drains the GPU being reassigned — allowing in-flight requests to complete before transitioning the pod to its new role. This avoids dropping active requests during transitions.

### Metrics Driving Autoscaling

The adaptive autoscaler monitors the following signals:

* **Prefill queue depth:** number of requests waiting for prefill. Rising depth signals insufficient prefill capacity.
* **Decode queue depth:** number of active decode sessions. Rising depth signals insufficient decode capacity.
* **TTFT (time to first token):** if TTFT exceeds the configured target, prefill is the bottleneck.
* **ITL (inter-token latency):** if ITL exceeds the configured target, decode is the bottleneck.
* **Per-GPU utilization:** compute utilization on prefill GPUs, memory bandwidth utilization on decode GPUs. Low utilization on either side may indicate over-provisioning.
* **KV cache transfer wait time:** time requests spend waiting for the transfer to complete. Rising wait times may indicate the transfer layer is saturated.

## 5. Hardware

### Tier 1: 4x RTX 4090 (PCIe)

* 24 GB VRAM per GPU, 96 GB total
* GPU-to-GPU communication over PCIe 4.0 at ~24 GB/s
* 2 GPUs allocated to prefill pods, 2 to decode pods
* Demonstrates the concept works even with slow interconnects
* Shows where KV cache transfer overhead becomes the bottleneck

### Tier 2: 2x A100 SXM4 (NVLink)

* 80 GB VRAM per GPU, 160 GB total
* GPU-to-GPU communication over NVLink at ~600 GB/s
* 1 GPU for prefill, 1 GPU for decode
* Demonstrates the architecture with minimal transfer overhead
* Shows the throughput gains when interconnect is not a bottleneck

### Model

Llama 3.1 8B or 13B in FP16, chosen to fit comfortably on both hardware tiers with ample VRAM remaining for KV cache on the decode GPU. Same model used across both tiers for fair comparison.

## 6. What We Benchmark

**Single request latency:** Disaggregated will be slightly slower due to KV cache transfer overhead. This is expected and acceptable.

**Throughput under load:** The primary metric. Disaggregated should outperform standard serving as concurrency increases, because each GPU is optimized for its specific phase with no interference.

**Scaling behavior:** As concurrent requests ramp from 1 to 100+, disaggregated should degrade more gracefully. The crossover point — where disaggregated overtakes standard serving — is the key finding.

**KV cache transfer overhead:** Measured in isolation across both hardware tiers. Demonstrates the cost of disaggregation and how interconnect bandwidth affects viability.

**GPU utilization:** Per-GPU compute and memory utilization over time. Standard serving should show fluctuating utilization as GPUs switch between phases. Disaggregated should show stable, high utilization on each GPU for its dedicated phase.

**Adaptive mode transitions:** Simulate a traffic pattern that starts low (unified mode), ramps to medium (static disaggregated), spikes high (dynamic rebalancing), and drops back down. Measure how quickly the scheduler transitions between modes and the impact on latency during transitions.

**Rebalancing effectiveness:** During a simulated traffic spike, measure how quickly the scheduler detects the need for more prefill or decode capacity, how long the GPU drain and reassignment takes, and whether TTFT and ITL recover to target levels after rebalancing.

## 7. Advantages of Disaggregated Serving

**Eliminates prefill-decode interference.** In standard serving, a new user's long prefill blocks token generation for existing users mid-conversation, causing latency spikes. Disaggregation removes this contention entirely. Prefill and decode never compete for the same GPU.

**Higher GPU utilization.** Each GPU runs the workload profile it handles best. Prefill GPUs stay compute-saturated. Decode GPUs stay memory-bandwidth-saturated. Neither GPU wastes cycles switching between fundamentally different workload types.

**Independent scaling.** Prefill and decode capacity can scale independently based on traffic patterns. During a spike of new users, add more prefill pods without touching decode. During periods of long conversations, add more decode capacity. Standard serving forces you to scale both together.

**Independent optimization.** Different parallelism strategies can be applied to each phase. Prefill might benefit from higher tensor parallelism for faster computation. Decode might benefit from less parallelism to maximize per-GPU KV cache capacity. Disaggregation allows tuning each phase without compromising the other.

**Hardware specialization potential.** Opens the door to running prefill on compute-optimized GPUs and decode on bandwidth-optimized hardware or specialized accelerators like Groq LPUs.

**Adaptive efficiency across all traffic levels.** The system is never in the wrong mode. At low traffic, it avoids disaggregation overhead entirely by running unified serving. At high traffic, it disaggregates and dynamically rebalances to maximize throughput. This eliminates the biggest con of disaggregated serving — that it performs worse at low concurrency — by simply not disaggregating when it doesn't help.

## 8. Disadvantages and Risks

**KV cache transfer overhead.** Every request pays the cost of moving the KV cache between GPUs. For long contexts (tens of thousands of tokens), the KV cache can be gigabytes. On slow interconnects like PCIe, this overhead can erase throughput gains entirely. This is the primary technical risk.

**Increased system complexity.** Two separate services, a transfer layer, routing logic, and additional failure modes. A prefill pod crash affects the entire pipeline. Standard serving is simpler to deploy, monitor, and debug.

**Single request latency is worse.** For any individual request, disaggregation adds transfer time on top of normal inference. The throughput gains only materialize under concurrent load. At low traffic, disaggregated serving is strictly worse than standard serving.

**Memory duplication.** Model weights must be loaded on both prefill and decode GPUs. In standard serving with tensor parallelism, weights are shared across GPUs. Disaggregation requires two separate copies of the model, reducing the total VRAM available for KV cache.

**Cold decode starts.** The decode GPU receives KV caches from conversations it has no prior context for. Every incoming cache is new. There is no locality benefit or warmup advantage.

**Crossover threshold.** Below a certain concurrency level, standard serving is faster and simpler. The project must identify this threshold precisely. The adaptive serving modes mitigate this risk by falling back to unified serving below the threshold, but the transition logic itself adds complexity.

**Mode transition overhead.** Switching between serving modes (unified to disaggregated, or rebalancing GPU assignments) requires draining in-flight requests and restarting pods in new roles. During transitions, capacity is temporarily reduced. Frequent oscillation between modes (if thresholds are poorly tuned) could cause instability. Hysteresis or cooldown periods are needed to prevent flapping.

## 9. Success Criteria

1. Disaggregated serving demonstrates measurably higher throughput than standard serving at a clearly identified concurrency threshold.
2. The crossover chart — showing where disaggregated overtakes standard serving — is produced with clear, reproducible data for both hardware tiers.
3. The GPU scheduler successfully places prefill and decode pods based on workload labels and GPU topology.
4. KV cache transfer latency is measured and documented for both NVLink and PCIe, showing the impact of interconnect bandwidth.
5. The adaptive autoscaler correctly transitions between unified, static disaggregated, and dynamic disaggregated modes based on traffic patterns.
6. Dynamic rebalancing responds to simulated traffic spikes within a measurable time window, and TTFT/ITL recover to target levels after rebalancing.
7. The project is documented with architecture diagrams, benchmark methodology, results, and analysis in a public GitHub repository.

## 10. Out of Scope

* Custom CUDA or Triton kernel development (separate project).
* Model training or fine-tuning.
* Multi-node distributed inference across separate machines.
* Production-grade fault tolerance, health checking, or auto-recovery.
* Integration with external load balancers or API gateways.
